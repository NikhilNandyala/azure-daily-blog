At 03:14 UTC on a Thursday, our monitoring system fired a P1 alert: twelve on-premises manufacturing sites had lost connectivity to Azure. The VMs running the factory floor integration services were unreachable from the corporate network. Production line reporting had stopped updating.

What made this incident genuinely difficult was the lie at the top of the stack: the ExpressRoute Gateway showed **BGP: Connected** in the Azure portal for the entire 47 minutes it took us to find the real root cause. Every status indicator said the circuit was healthy. It wasn't.

This is the full RCA.

---

## Architecture Overview

Our hybrid connectivity uses dual ExpressRoute circuits — one primary (West US 2) and one failover (East US) — both terminated on the same ExpressRoute Gateway in the primary region. On-premises, we run two Cisco ASR 1001-X routers (one per circuit) in BGP AS 65010. Azure runs AS 65515.

<DiagramImage
  src="https://res.cloudinary.com/ybcsc7rq/image/upload/f_auto,q_auto/Hybrid_ExpressRoute_architecture_diagram_fhljiq"
  alt="Hybrid connectivity architecture: on-premises AS 65010 with dual Cisco ASR routers connecting via two ExpressRoute circuits to an ExpressRoute Gateway in West US 2, peered to Hub VNet with spoke VNets"
  caption="Pre-incident architecture: dual ExpressRoute circuits, active-active BGP sessions, Hub-Spoke VNet layout"
/>

**Normal traffic flow:**
- Twelve on-prem sites connect to the primary ASR via MPLS
- Primary circuit (West US 2): active, preferred path via BGP MED
- Failover circuit (East US): hot-standby, accepts traffic within 30 seconds if primary fails
- Hub VNet (10.0.0.0/16) peers to six spoke VNets via VNet peering
- ExpressRoute Gateway: `ErGw1AZ` SKU, Zone-Redundant (post-incident change; was `Standard` before)

Under normal conditions, the gateway processes ~800 BGP prefixes: 620 from on-prem and 180 Azure routes propagated back to the CE routers.

## Alert and Initial Symptoms

**03:14 UTC — Alert fires:**

```
ALERT [P1]: Cross-premises connectivity failure
Scope: 12 sites × manufacturing integration VMs
Metric: ExpressRouteGatewayPacketsPerSecond = 0
Duration: 3 minutes (threshold: 2 minutes)
```

The `ExpressRouteGatewayPacketsPerSecond` metric dropping to zero is unambiguous — no data-plane packets flowing through the gateway. The alert was configured after a near-miss six months earlier when a misconfigured route filter silently dropped selective traffic for four hours before anyone noticed.

**What the portal showed at 03:15:**

| Component | Status shown |
|-----------|-------------|
| ExpressRoute Circuit (primary) | **Provisioned — ✓** |
| ExpressRoute Circuit (failover) | **Provisioned — ✓** |
| ExpressRoute Gateway | **Succeeded** |
| BGP peer 10.10.1.1 (primary CE) | **Connected** |
| BGP peer 10.10.1.5 (failover CE) | **Connected** |
| Gateway health probe | **Healthy** |

Everything green. Nothing in the Activity Log. No resource health events. No Service Health alerts for West US 2.

This is the situation that costs you the most time: all the leading indicators say everything is fine, but production is down.

## Phase 1 — Confirming the Scope

Before touching anything, we confirmed what was and wasn't working.

**From on-premises (CE router):**

```
ASR1001X-primary# show bgp summary
Neighbor        V    AS MsgRcvd MsgSent   TblVer  InQ OutQ  Up/Down State/PfxRcd
10.0.0.4        4 65515    4821    4819     2841    0    0  2d09h        188
10.0.0.5        4 65515    4820    4818     2841    0    0  2d09h        188
```

BGP sessions to Azure's two gateway IPs were both up, with the expected prefix counts. Hold timers refreshing normally. From the on-premises side, BGP was completely healthy.

**Reachability test from on-prem:**

```bash
# Ping Azure spoke VM
ping 10.1.4.22 repeat 100 timeout 2
# Result: !!!!!!!!!!!!! (12 successes, 88 timeouts)
# Intermittent — not total packet loss
```

Intermittent, not total. That shifted the diagnosis away from a BGP session being down (which would be total loss) toward a routing plane issue.

**From Azure (checking effective routes on a spoke VM NIC):**

```bash
az network nic show-effective-route-table \
  --resource-group rg-spoke-manufacturing \
  --name nic-integration-vm-01 \
  --output table | grep "10.10\|10.20\|10.30"
```

```
Source    State    AddressPrefix    NextHopType       NextHopIP
-------   ------   -------------    -----------       ---------
(empty)
```

No on-premises routes. The spoke VM's NIC effective route table had no entries for the 10.10.x.x, 10.20.x.x, or 10.30.x.x on-premises supernets. That's the smoking gun: Azure VMs had no route to on-premises, despite the BGP sessions showing healthy.

## Phase 2 — Gateway Diagnostics

**03:31 UTC — Checking what the gateway actually knows:**

```bash
# What routes has the gateway learned from on-prem?
az network vnet-gateway list-learned-routes \
  --resource-group rg-hub-westus2 \
  --name er-gateway-hub \
  --output json | jq 'length'
```

Output: `0`

The ExpressRoute Gateway had learned **zero routes** from the on-premises BGP peers — despite both BGP sessions showing as Connected. This is the central contradiction of the incident. A BGP session can be in the Established state (keepalives exchanging, hold timer refreshing) while the **route table** is completely empty due to a control-plane processing failure.

```bash
# Check the BGP peer status from Azure's side
az network vnet-gateway list-bgp-peer-status \
  --resource-group rg-hub-westus2 \
  --name er-gateway-hub \
  --output table
```

```
Neighbor    ASN    State        ConnectedDuration    RoutesReceived    MessagesSent    MessagesReceived
----------  -----  -----------  -------------------  ----------------  --------------  ----------------
10.10.1.1   65010  Connected    2.09:14:32           0                 4819            4821
10.10.1.5   65010  Connected    2.09:13:58           0                 4818            4820
```

`RoutesReceived: 0` on both peers. The gateway control plane had accepted the TCP sessions and was exchanging keepalives but had not processed a single UPDATE message from either CE router. The gateway was deaf to BGP UPDATE packets while still participating in the BGP session.

**Checking gateway resource health:**

```bash
az resource show \
  --ids "/subscriptions/${SUBSCRIPTION_ID}/resourceGroups/rg-hub-westus2/providers/Microsoft.Network/virtualNetworkGateways/er-gateway-hub" \
  --query "properties.provisioningState"
```

Output: `"Succeeded"`

```bash
# Check recent operations
az monitor activity-log list \
  --resource-id "/subscriptions/${SUBSCRIPTION_ID}/resourceGroups/rg-hub-westus2/providers/Microsoft.Network/virtualNetworkGateways/er-gateway-hub" \
  --start-time "2026-06-04T03:00:00Z" \
  --output table
```

No operations. The gateway hadn't been modified. No platform operations. No scale events. The control-plane failure was silent — Azure's management plane considered the gateway healthy while the data plane control processor had entered a failed state.

## Phase 3 — BGP Troubleshooting

We queried the gateway diagnostics logs to confirm the UPDATE processing failure.

<DiagramImage
  src="https://res.cloudinary.com/ybcsc7rq/image/upload/f_auto,q_auto/Gateway_failure_troubleshooting_flowchart_ahfqyd"
  alt="Gateway failure troubleshooting flowchart showing 5 decision steps: BGP session state check, route count check, UPDATE message processing check, control-plane health check, and failover decision"
  caption="The 5-step diagnostic flowchart we ran. The BGP session state check (step 1) passed — that's what added 47 minutes to the incident."
/>

**KQL — BGP UPDATE processing in gateway diagnostic logs:**

```kusto
AzureDiagnostics
| where ResourceType == "EXPRESSROUTEGATEWAYS"
| where Category == "GatewayDiagnosticLog"
| where TimeGenerated > ago(2h)
| where Message contains "BGP" or Message contains "route"
| project TimeGenerated, OperationName, Message
| order by TimeGenerated desc
```

The query returned keepalive logs going back two days (normal) but **no UPDATE RECEIVED entries** after `2026-06-01T18:23:41Z` — three days before the incident. The gateway had stopped processing UPDATE messages 72 hours before the alert fired, but the route table corruption wasn't noticed because the routes were already programmed from the last successful UPDATE cycle and remained in the routing table until the gateway performed an internal refresh at 03:11 UTC — at which point it cleared its stale route table and couldn't repopulate it.

The BGP route flow under normal operation:

<DiagramImage
  src="https://res.cloudinary.com/ybcsc7rq/image/upload/f_auto,q_auto/BGP_route_flow_diagram_ycu8za"
  alt="BGP route flow diagram showing the 5-step sequence: OPEN message exchange establishing the session, KEEPALIVE exchange, UPDATE messages carrying prefixes, ACK from peer, and INSTALL into the FIB"
  caption="BGP route programming flow. The gateway was completing steps 1–3 (session, keepalive) but failing silently at step 4 — UPDATE processing. Routes were never installed into the FIB."
/>

In our incident, the gateway was executing steps 1 through 3 successfully (the BGP TCP session was up and keepalives were exchanging). The failure was at step 4: the control-plane processor responsible for parsing UPDATE message TLVs and installing routes into the Forwarding Information Base (FIB) had entered a failed state. Azure's management plane health checks poll the BGP session state and the gateway's VM-level health — neither of which catches a FIB programming failure.

---

## Root Cause

**Control-plane route table corruption following a platform firmware update.**

Azure's platform team later confirmed via support case that the ExpressRoute Gateway's non-zone-redundant SKU (`Standard`) had received a firmware update on June 1 at 18:23 UTC. The update completed successfully on the primary gateway instance but left the BGP route processor in a degraded state where it accepted TCP-level BGP session establishment and keepalive traffic but silently dropped UPDATE packets without logging the failure or surfacing it via the resource health API.

The degraded state persisted until 03:11 UTC on June 4 when the gateway's internal route refresh timer fired. The refresh cleared the stale route table (which had been programmed from the last successful UPDATE on June 1) and attempted to re-learn all routes from scratch. With the UPDATE processor still degraded, the re-learn produced an empty routing table. The effective routes on every spoke VNet NIC went to zero. Traffic dropped.

**Why BGP showed Connected:** The BGP TCP session and keepalive functions run on a separate process from the UPDATE handler. The keepalive process was healthy; the UPDATE handler was not. Azure's health model polled the session state, not the FIB state.

**Why it took 47 minutes to find:** The BGP session status is the first thing every network engineer checks. It showed Connected. Every subsequent check that confirmed BGP was working (CE router BGP summary, UPDATE message counters) was directionally misleading because the failure was on the Azure side of the path, not the circuit or CE router side. We didn't check `list-learned-routes` until Phase 2, 17 minutes into the incident.

## Decision to Fail Over

**03:47 UTC — Failover decision:**

At this point we had confirmed:
- On-prem BGP healthy (CE routers normal)
- ExpressRoute circuits healthy (provisioned, no alarms)
- Gateway management plane healthy (provisioningState: Succeeded)
- Gateway control plane broken (0 learned routes despite Connected peers)
- Root cause: gateway internal failure, not fixable via configuration

Options considered:

| Option | Time estimate | Risk |
|--------|--------------|------|
| Gateway reset (soft restart) | ~5 min | May not fix firmware-level failure |
| Gateway delete + recreate | ~45 min | Long outage extension |
| **Regional failover to East US** | **~25 min** | Low — failover circuit already provisioned |
| Wait for Azure support | Unknown | Unacceptable for P1 |

We chose regional failover. The failover ExpressRoute circuit had been provisioned for exactly this scenario: a gateway-level failure in the primary region that couldn't be fixed quickly. The decision was made at 03:47 — 33 minutes into the incident.

## Failover Execution

**03:49 UTC — Start failover:**

The failover circuit connected to an ExpressRoute Gateway in East US (`er-gateway-eastus`, `Standard` SKU). The Hub VNet in East US was already peered to the critical spoke VNets via Global VNet Peering.

**Step 1 — Confirm failover gateway is healthy:**

```bash
az network vnet-gateway list-bgp-peer-status \
  --resource-group rg-hub-eastus \
  --name er-gateway-eastus \
  --output table
```

```
Neighbor    ASN    State        RoutesReceived
----------  -----  -----------  ----------------
10.10.2.1   65010  Connected    621
```

621 routes received — East US gateway was healthy and had a full routing table.

**Step 2 — Prepend AS path on primary CE to drain traffic toward East US:**

Rather than cutting over the circuit, we used BGP AS path prepending on the primary CE router to make the failed gateway's path less preferred, forcing on-premises traffic to shift to the failover circuit.

```
ASR1001X-primary# conf t
ASR1001X-primary(config)# route-map AZURE-OUT permit 10
ASR1001X-primary(config-route-map)# set as-path prepend 65010 65010 65010
ASR1001X-primary(config-route-map)# exit
ASR1001X-primary(config)# router bgp 65010
ASR1001X-primary(config-router)# neighbor 10.0.0.4 route-map AZURE-OUT out
ASR1001X-primary(config-router)# neighbor 10.0.0.5 route-map AZURE-OUT out
ASR1001X-primary(config-router)# end
ASR1001X-primary# clear ip bgp 10.0.0.4 soft out
ASR1001X-primary# clear ip bgp 10.0.0.5 soft out
```

The prepend didn't help Azure accept routes (the UPDATE processor was broken) but it told on-premises traffic to prefer the East US path for return traffic symmetry.

**Step 3 — Force Azure to use East US path for VNet-to-on-prem traffic:**

```bash
# Add a route table on the Hub VNet GatewaySubnet to force traffic to East US
# (This is a temporary override while primary gateway is broken)
az network route-table create \
  --resource-group rg-hub-westus2 \
  --name rt-force-eastus-failover \
  --disable-bgp-route-propagation true

az network route-table route create \
  --resource-group rg-hub-westus2 \
  --route-table-name rt-force-eastus-failover \
  --name route-onprem-10-10 \
  --address-prefix "10.10.0.0/16" \
  --next-hop-type VirtualNetworkGateway

# Apply to spoke VNet subnets that host integration VMs
az network vnet subnet update \
  --resource-group rg-spoke-manufacturing \
  --vnet-name vnet-spoke-manufacturing \
  --name snet-integration \
  --route-table rt-force-eastus-failover
```

Wait — this doesn't point to the East US gateway directly. The VirtualNetworkGateway next-hop on a spoke VNet uses the Hub VNet's gateway via peering. Since the Hub VNet is peered to both West US 2 and East US Hubs, we needed to update the peering to use the East US Hub's gateway.

**Correct Step 3 — Update VNet peering to use East US hub gateway:**

```bash
# Update spoke VNet peering to use East US hub as gateway transit
az network vnet peering update \
  --resource-group rg-spoke-manufacturing \
  --vnet-name vnet-spoke-manufacturing \
  --name peer-to-hub-westus2 \
  --set useRemoteGateways=false

az network vnet peering create \
  --resource-group rg-spoke-manufacturing \
  --vnet-name vnet-spoke-manufacturing \
  --name peer-to-hub-eastus-failover \
  --remote-vnet /subscriptions/${SUBSCRIPTION_ID}/resourceGroups/rg-hub-eastus/providers/Microsoft.Network/virtualNetworks/vnet-hub-eastus \
  --allow-vnet-access true \
  --use-remote-gateways true
```

**04:09 UTC — Connectivity confirmed restored:**

```
ping 10.1.4.22 repeat 100 timeout 2
# Result: !!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!
# 100/100 success
```

Total failover time from decision to restored: **22 minutes.**
Total incident duration: **55 minutes** (03:14 → 04:09).

## Gateway Rebuild

With production restored via the East US failover, we turned to the West US 2 gateway.

**04:30 UTC — Gateway reset (first attempt):**

```bash
az network vnet-gateway reset \
  --resource-group rg-hub-westus2 \
  --name er-gateway-hub
```

This resets the gateway (causes a ~30-second downtime cycle on non-zone-redundant gateways). After the reset, we checked route learning:

```bash
az network vnet-gateway list-learned-routes \
  --resource-group rg-hub-westus2 \
  --name er-gateway-hub \
  --output json | jq 'length'
```

Output: `0` — still broken after reset.

**05:15 UTC — Gateway delete and recreate:**

The reset did not fix the firmware-level corruption. We deleted and recreated the gateway, taking the opportunity to upgrade from `Standard` to `ErGw1AZ` (Zone-Redundant).

```bash
# Delete the broken gateway (traffic is on East US path, so no additional outage)
az network vnet-gateway delete \
  --resource-group rg-hub-westus2 \
  --name er-gateway-hub

# Recreate with Zone-Redundant SKU
az network vnet-gateway create \
  --resource-group rg-hub-westus2 \
  --name er-gateway-hub \
  --location westus2 \
  --vnet vnet-hub-westus2 \
  --gateway-type ExpressRoute \
  --sku ErGw1AZ \
  --public-ip-address pip-er-gateway-hub-1 pip-er-gateway-hub-2 \
  --no-wait
```

The `ErGw1AZ` SKU deploys gateway instances across Azure Availability Zones. A zone failure cannot take out the gateway — at least one instance is always available in a different zone. The firmware-level failure we experienced would have been caught by Azure's zone-health monitoring had we been on a zone-redundant SKU.

Gateway recreation completed at **06:47 UTC.** ExpressRoute connections were re-attached, BGP reconverged, and we validated full route learning:

```bash
az network vnet-gateway list-learned-routes \
  --resource-group rg-hub-westus2 \
  --name er-gateway-hub \
  --output json | jq 'length'
# Output: 621
```

We then reverted the spoke VNet peering to use the rebuilt West US 2 gateway and removed the East US failover peering.

## Alerting Added After the Incident

The `ExpressRouteGatewayPacketsPerSecond = 0` alert fired correctly. But it fired 47 minutes into a 3-day silent degradation. We added two new alerts to catch the pre-failure state earlier:

**Alert 1 — Zero routes learned (catches the exact failure mode):**

```bash
az monitor metrics alert create \
  --resource-group rg-hub-westus2 \
  --name "ER-GW-ZeroLearnedRoutes" \
  --scopes "/subscriptions/${SUBSCRIPTION_ID}/resourceGroups/rg-hub-westus2/providers/Microsoft.Network/virtualNetworkGateways/er-gateway-hub" \
  --condition "avg ExpressRouteBgpPeerRouteCount < 100" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --severity 1 \
  --description "ER Gateway learned fewer than 100 BGP routes — possible control-plane failure"
```

Note: `ExpressRouteBgpPeerRouteCount` is a metric available in Azure Monitor for ExpressRoute Gateways. A threshold of 100 gives a comfortable buffer below our normal 621 while catching a near-zero state immediately.

**Alert 2 — Log Analytics: BGP UPDATE absence detection:**

```kusto
// Alert if no BGP UPDATE RECEIVED log entry in last 30 minutes
// when the BGP session is Connected
AzureDiagnostics
| where ResourceType == "EXPRESSROUTEGATEWAYS"
| where Category == "GatewayDiagnosticLog"
| where TimeGenerated > ago(30m)
| where Message contains "UPDATE RECEIVED"
| summarize UpdateCount = count()
| where UpdateCount == 0
```

This query, run as a Log Analytics scheduled alert every 30 minutes during business hours, would have caught the June 1 failure on June 1 — not three days later.

## High Availability Recommendations

Based on this incident, here's the HA architecture we implemented:

<DiagramImage
  src="https://res.cloudinary.com/ybcsc7rq/image/upload/f_auto,q_auto/Final_high-availability_reference_architecture_diagram_oljlgs"
  alt="Final high-availability reference architecture showing two zone-redundant ExpressRoute Gateways (West US 2 and East US) with dual ExpressRoute circuits, global VNet peering between hubs, and a shared services layer"
  caption="Post-incident HA reference architecture: zone-redundant gateways in two regions, dual circuits per region, automatic BGP health monitoring"
/>

**Change 1 — Zone-Redundant Gateway SKU (`ErGw1AZ`)**

The `Standard` SKU runs both gateway instances in the same fault domain. A firmware failure affecting that fault domain takes out both instances simultaneously. `ErGw1AZ` distributes instances across Availability Zones — a zone failure or zone-scoped firmware issue cannot take out the gateway.

Cost delta: approximately $220/month more than `Standard` in West US 2. Worth it.

**Change 2 — BGP route count metric alert**

`ExpressRouteBgpPeerRouteCount < 100` alerts within 5 minutes of the route table going to near-zero. This alert would have fired at 03:11 UTC in our incident — 3 minutes after the route table cleared — giving us a direct signal instead of the indirect `PacketsPerSecond = 0` alert that fired 3 minutes later.

**Change 3 — Automated BGP UPDATE audit**

A Log Analytics scheduled alert query (shown above) runs every 30 minutes and fires if no BGP UPDATE RECEIVED log entries appear in the last 30 minutes while the BGP session is up. This catches the silent degradation before it becomes an outage.

**Change 4 — Pre-tested failover runbook**

We documented and tested the full failover procedure (the commands in the Failover Execution section above) as an Azure Runbook. The runbook can be triggered from an alert action group. Target: failover in under 10 minutes via automation instead of 22 minutes of manual CLI.

**Change 5 — Global VNet Peering pre-provisioned**

The Global VNet Peering between West US 2 Hub and East US Hub now exists permanently (not provisioned during the incident). Spoke VNet peering changes are the only manual steps needed during failover.

## Prevention Checklist

| Check | Action |
|-------|--------|
| Gateway SKU | Use `ErGw1AZ`, `ErGw2AZ`, or `ErGw3AZ` — never `Standard` for production |
| BGP route count alert | Alert when `ExpressRouteBgpPeerRouteCount < 50%` of expected baseline |
| BGP UPDATE audit | Log Analytics scheduled query every 30 min during business hours |
| Failover circuit | Pre-provisioned hot-standby circuit in a second region |
| Failover runbook | Tested automation that can complete failover in < 10 minutes |
| Peering pre-provisioned | Global VNet Peering between Hub VNets in both regions always active |
| Gateway diagnostic logs | Enable `GatewayDiagnosticLog` on all ExpressRoute Gateways |

## Lessons Learned

**1. BGP session state is not a health signal for the data plane.**

A BGP session can be in the Established state while the route programming pipeline is completely broken. `show bgp summary` on the CE router and the BGP peer status in Azure both said healthy for 47 minutes while zero packets were being forwarded. Design your monitoring to check **learned routes** and **effective routes**, not just session state.

**2. The route count metric is your early warning system.**

`ExpressRouteBgpPeerRouteCount` dropping from 621 to 0 is an unambiguous signal. The `PacketsPerSecond` metric dropping to zero is a consequence — it tells you the outage has started, not that it's about to start. Monitor the cause, not just the effect.

**3. Gateway reset does not fix firmware-level failures.**

`az network vnet-gateway reset` is the right first action for many gateway issues (stuck BGP sessions, transient VPN disconnects). It did not help here because the failure was in the firmware layer, not the BGP process. Know when to skip to recreate.

**4. Zone-redundant SKUs are not optional for production ExpressRoute.**

The `Standard` SKU's single fault domain is exactly the attack surface this firmware failure exploited. `ErGw1AZ` would not have prevented the firmware update from reaching one zone, but Azure's zone-health monitoring would have detected the unhealthy zone and shifted traffic to a healthy zone instance, probably before the incident even surfaced.

**5. Failover circuits must be tested, not just provisioned.**

Our failover circuit had been provisioned and BGP was configured, but we had never executed the peering change procedure under pressure. It took 22 minutes. With the runbook, pre-provisioned peering, and one practice run, we now target 8 minutes.

**6. Silent failures need active detection.**

This failure was silent for 72 hours before it surfaced. The only reason it surfaced at all was the internal route refresh timer. Without the BGP UPDATE audit query, this category of failure is invisible to standard monitoring until production drops. Add the audit query — it costs almost nothing and catches a failure mode that Azure's built-in health APIs currently do not surface.
